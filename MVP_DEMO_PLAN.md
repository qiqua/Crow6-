# AI 编程桌面产品开发测试版设计方案

## 1. 版本定位

本版本是面向投资人、早期用户和内部验证的开发测试版，不追求完整商业化能力，也不追求企业级安全能力。

核心目标是：

- 快速做出一个可演示、可下载、可体验的 AI 编程桌面产品。
- 展示“自然语言生成软件”的核心能力。
- 展示“代码库感知 + AI 修改代码 + Diff 审查”的产品闭环。
- 展示“多 Agent 协同”的概念和基础体验。
- 展示未来可以扩展为 Skills 市场、多模型生态和企业平台的空间。

该版本重点服务于融资展示，因此优先级是：

```text
演示效果 > 产品闭环 > 技术可扩展性 > 安全完整性 > 商业化完整性
```

### 1.1 对标演示版本目标

开发测试版的目标不是完整复刻某个产品的内部实现，而是在测试和路演时做到“演示观感、核心路径和用户感知能力”高度一致。

对标目标：

- 有一个可启动的桌面客户端。
- 首页和主界面具备完整产品感。
- 用户能打开本地项目。
- AI 能理解项目结构并引用文件。
- AI 能根据自然语言修改代码。
- 修改前能展示 Diff。
- 用户点击后能应用修改。
- 多 Agent 面板能展示任务推进过程。
- Skills 页面能展示未来生态入口。
- 模型选择、设置页、任务日志等细节要让产品看起来完整。

不做的事情：

- 不复制对方品牌、文案、Logo、素材和专有 UI。
- 不追求完全一样的底层 Agent 实现。
- 不做完整商业后台和企业安全体系。
- 不把不稳定的大功能放进现场主 Demo。

Demo 一致性的定义：

```text
用户看到的是一个完整的 AI 编程桌面产品，
能完成“打开项目 → 理解代码 → 修改代码 → 展示 Diff → 应用修改 → 展示 Agent 协作”的闭环。
```

### 1.2 演示一致性矩阵

为了让测试版本看起来接近成熟演示版本，Alpha 需要优先还原“用户能看到和感受到的能力”，而不是优先还原复杂底层系统。

| 演示观感 | Alpha 实现方式 | 是否必须真实完整 |
| --- | --- | --- |
| 桌面客户端 | Electron 可运行应用，有完整主界面 | 必须 |
| 多 Agent 协同 | 顺序状态机 + 多 Agent 面板 + 阶段输出 | 不需要真实并发 |
| 代码库感知 | 文件树 + 关键词搜索 + 项目摘要 + 文件引用 | 必须可用 |
| 自动修改代码 | 局部 patch + Diff Viewer + Apply | 必须可用 |
| Skills 市场 | 内置 Skill 卡片 + Prompt 模板执行 | 不需要云端市场 |
| 模型生态 | OpenAI-compatible 配置 + 模型选择器 | 不需要 20+ 厂商 |
| 终端能力 | 命令建议 + 复制命令 + 可选单次执行 | 不需要完整终端 |
| 项目生成 | 内置模板 + AI 局部定制 | 不需要从零生成 |
| 用户中心 | 本地模式 / 假登录状态 | 不需要真实账号 |
| 更新日志 | About / Changelog 静态页面 | 不需要自动更新 |

核心原则：

```text
演示层要像完整产品，底层实现可以是 Alpha 简化版。
```

### 1.3 测试版必须避免的演示问题

以下问题会明显破坏融资演示观感，开发时要优先避免：

- 打开项目后长时间无反馈。
- AI 回答没有文件路径引用。
- Diff 太大，看起来像整文件被重写。
- Apply 后页面没有可见变化。
- Agent 面板只是静态装饰，和任务过程不同步。
- 模型响应失败后没有兜底提示。
- Skills 页面只有空壳，没有可点击效果。
- 设置页无法配置模型。
- App 看起来像 Demo 原型，而不是桌面产品。

## 3. 核心演示卖点

开发测试版只需要打透 5 个卖点。

### 3.1 一句话生成项目

用户输入：

```text
帮我生成一个带登录页、仪表盘和用户列表的 React 管理后台
```

系统展示：

- AI 自动识别需求类型。
- 从内置模板中选择最接近的项目模板。
- 基于用户描述调整页面、文案、颜色和模块。
- 写入核心文件。
- 提示安装依赖或复制启动命令。
- 打开预览说明。

演示价值：

- 投资人能直接看到“想法变软件”。
- 这是最容易被理解的场景。

Alpha 版本实现原则：

```text
模板生成优先，AI 定制为辅。
```

不建议现场完全依赖模型从零生成大量文件。融资演示时应使用稳定模板兜底。

---

### 3.2 读懂已有代码库

用户打开一个已有项目后提问：

```text
这个项目的登录逻辑在哪里？帮我总结一下数据流。
```

系统展示：

- 扫描项目目录。
- 找到相关文件。
- 给出路径引用。
- 总结调用关系。

演示价值：

- 证明产品不是普通聊天机器人。
- 强调“代码库感知”。

---

### 3.3 AI 自动修改代码

用户输入：

```text
把登录按钮改成渐变色，并增加加载状态。
```

系统展示：

- AI 定位相关文件。
- 生成修改方案。
- 展示 Diff。
- 用户点击应用。
- 页面变化。

演示价值：

- 打通 AI 编程助手核心闭环。
- 很适合现场演示。

---

### 3.4 多 Agent 协同面板

用户输入：

```text
帮我给这个项目增加用户管理模块，包括页面、接口 mock 和基础测试。
```

系统展示 Agent 面板：

```text
Planner Agent：正在拆解任务
Codebase Agent：正在分析项目结构
Engineer Agent：正在生成代码
Reviewer Agent：正在检查修改
Tester Agent：正在生成测试建议
```

实际 MVP 可以先做“半真实多 Agent”：

- Planner 真实调用模型。
- Codebase Agent 真实搜索代码。
- Engineer Agent 真实生成 patch。
- Reviewer / Tester 可以先做轻量实现或模板化输出。

演示价值：

- 建立和普通 AI 编辑器的差异化。
- 投资人更容易理解平台想象空间。

---

### 3.5 Skills 概念展示

开发测试版不需要完整市场，只需要本地 Skills 页面。

展示几个内置 Skill：

- 代码审查
- 生成测试
- 一键生成 README
- 生成接口文档
- 生成部署配置
- UI 美化

用户点击 Skill 后，自动填入对应工作流提示词并执行。

演示价值：

- 展示未来生态扩展能力。
- 给投资人想象“应用市场”和“企业私有技能”。

## 4. 开发测试版功能范围

### 4.1 必须做

| 模块 | 功能 | 说明 |
| --- | --- | --- |
| 桌面端 | Electron 应用壳 | 可安装、可启动、有品牌感 |
| 项目管理 | 打开本地目录 | 支持读取文件树 |
| AI 聊天 | 基础对话 | 支持流式输出 |
| 文件工具 | 读取文件、搜索文件 | 支撑代码库问答 |
| 代码修改 | Patch 优先 + Diff 应用 | 核心演示能力 |
| 模型接入 | OpenAI-compatible API | 兼容 DeepSeek / Qwen / OpenAI 网关 |
| 多 Agent 面板 | 展示任务过程 | 可半真实实现 |
| Skills 页面 | 内置 5-8 个技能 | 先不做云端市场 |
| 模板项目 | 内置 3-5 个演示模板 | 保证一句话生成项目稳定 |
| 设置页 | 配置 API Key 和模型 | 支持最少模型配置 |
| 本地日志 | 记录任务过程 | 方便调试和演示 |

---

### 4.2 可以简化

| 模块 | 简化方式 |
| --- | --- |
| 用户系统 | Alpha 版本不做登录，或只做本地假登录 |
| 订阅计费 | 不做 |
| 云端后端 | 暂不做，或者只做官网/下载页 |
| 自动更新 | 不做，手动发安装包 |
| 企业权限 | 不做 |
| Skill 市场 | 只做本地内置 Skills |
| 向量数据库 | 第一版可先不用，使用关键词搜索 + 文件摘要 |
| 完整 LSP | 暂不做，后续再接 |
| 真实终端 | 默认不做完整 xterm.js，只做命令建议和任务日志 |
| 复杂沙箱 | 暂不做 |
| 插件签名 | 暂不做 |

### 4.3 暂不做

开发测试版不建议做：

- 完整云端账号体系。
- 团队协作空间。
- 订阅计费。
- 企业私有化部署。
- 完整插件市场。
- Skill 签名和审核。
- 复杂权限矩阵。
- 完整 Agent 并发调度系统。
- 跨项目长期记忆。
- 复杂自动化测试平台。

这些能力适合放在融资材料的 Roadmap 里，不适合 Alpha 阶段消耗开发周期。

### 4.4 对标演示版的最小功能包

如果目标是测试时尽量接近成熟演示版本，最小功能包应固定为以下内容。

P0 必做：

- 桌面应用启动页。
- 主工作区三栏布局。
- 项目目录选择。
- 文件树展示。
- 文件内容预览。
- AI Chat 流式输出。
- 模型设置页。
- 代码搜索。
- 项目结构摘要。
- 文件路径引用。
- Agent Console 状态机。
- 局部 patch 生成。
- Diff Viewer。
- Apply 修改。
- 本地任务日志。

P1 做成展示即可：

- Skills 页面。
- Changelog 页面。
- 模型生态选择器。
- 命令建议面板。
- 项目模板列表。
- 假登录 / 本地用户状态。

P2 暂时只放 Roadmap：

- 云端 Skill 市场。
- 真实用户系统。
- 订阅计费。
- 企业私有化。
- 完整终端。
- 完整语义索引。
- 完整多 Agent 并发调度。

## 5. 简化技术架构

开发测试版采用本地优先架构，不依赖复杂云后端。

```text
┌───────────────────────────────────────┐
│           Electron Desktop             │
│                                       │
│  React UI                             │
│  Chat Panel                           │
│  Agent Console                        │
│  File Explorer                        │
│  Diff Viewer                          │
│  Skills Page                          │
│  Settings Page                        │
└───────────────────┬───────────────────┘
                    │
                    ▼
┌───────────────────────────────────────┐
│          Local Node Runtime            │
│                                       │
│  File Reader                          │
│  File Writer                          │
│  Code Search                          │
│  Patch Manager                        │
│  Command Suggestion                    │
│  Agent Orchestrator                   │
│  Skill Runtime                        │
│  Template Engine                      │
└───────────────────┬───────────────────┘
                    │
                    ▼
┌───────────────────────────────────────┐
│           Model Adapter                │
│                                       │
│  OpenAI-compatible API                │
│  DeepSeek                             │
│  Qwen                                 │

└───────────────────────────────────────┘
```

Alpha 版本不需要独立后端，所有核心能力先在桌面端和本地 Runtime 完成。

模型范围说明：

- Alpha 默认只实现 OpenAI-compatible 协议。
- DeepSeek、Qwen、Moonshot、OpenRouter、SiliconFlow 都通过同一套 OpenAI-compatible 配置接入。
- 暂不单独适配 Claude、Gemini、Ollama 原生协议。
- 如果需要 Ollama，只支持其 OpenAI-compatible endpoint，不做额外适配。

## 6. 推荐技术栈

### 6.1 桌面端

```text
Electron
React
TypeScript
Vite
TailwindCSS
shadcn/ui
Zustand
TanStack Query
Monaco Editor
```

说明：

- Alpha 核心版不强依赖 xterm.js。
- 底部区域优先做 Logs / Task Timeline。
- 命令执行先做“复制命令”或“确认后执行一次命令”，不做完整交互终端。

### 6.2 本地 Runtime

```text
Node.js
chokidar
fast-glob
ignore
simple-git
execa
jsdiff / diff
```

说明：

- `execa` 仅用于明确确认后的单次命令执行。
- Alpha 阶段不做长期运行的交互式终端会话管理。

### 6.3 模型接入

Alpha 版本优先支持 OpenAI-compatible 接口。

这样可以兼容：

- OpenAI
- DeepSeek
- Qwen
- Moonshot
- OpenRouter
- SiliconFlow
- 本地兼容服务

后续再单独适配 Claude、Gemini、Ollama 原生协议。

### 6.4 数据存储

Alpha 版本使用本地文件即可：

```text
~/.crow6/config.json
~/.crow6/tasks/
~/.crow6/logs/
~/.crow6/skills/
~/.crow6/templates/
```

暂不需要 PostgreSQL、Redis、向量数据库。

API Key 存储建议：

- 优先使用系统凭据管理器。
- Windows 使用 Credential Manager。
- macOS 使用 Keychain。
- 如果 Alpha 阶段为了开发速度使用本地配置文件，需要在设置页明确提示“API Key 仅存储在本机测试环境”。

## 7. 页面设计

### 7.1 主界面

布局建议：

```text
┌────────────────────────────────────────────────────────┐
│ Top Bar：项目名 / 模型选择 / 运行状态 / 设置             │
├───────────────┬──────────────────────┬─────────────────┤
│ File Explorer │     Editor / Diff     │    AI Panel     │
│               │                      │                 │
│ 项目文件树     │  文件预览 / 修改对比   │  Chat / Agent   │
│               │                      │                 │
├───────────────┴──────────────────────┴─────────────────┤
│ Logs / Task Timeline / Command Suggestions              │
└────────────────────────────────────────────────────────┘
```

重点：

- UI 要有“未来感”和“产品完成度”。
- 即使功能不完整，也要让界面显得成熟。
- 多 Agent 面板要可视化，适合演示。

界面细节建议：

- 顶部要有产品名、当前项目名、模型选择器、运行状态、设置按钮。
- 左侧文件树要展示真实项目结构。
- 中间区域默认显示欢迎页，选择文件后显示代码预览，生成修改后切换为 Diff。
- 右侧 AI Panel 需要支持 Chat 和 Agent 两个 Tab。
- 底部区域展示任务日志、命令建议和最近工具调用。
- 设置页至少包含 API Base URL、API Key、模型名三个字段。
- About 页面展示版本号、更新日志和 Desktop 下载感。

视觉目标：

```text
不要像脚手架 Demo，要像一个已经准备内测的桌面产品。
```

### 7.2 AI Chat Panel

功能：

- 输入自然语言任务。
- 展示流式回答。
- 展示 AI 正在读取哪些文件。
- 展示 AI 准备修改哪些文件。
- 展示 Apply 按钮。

建议交互：

```text
用户输入任务
  → AI 输出计划
  → AI 搜索相关文件
  → AI 生成修改
  → 展示 Diff
  → 用户点击 Apply
```

### 7.3 Agent Console

Agent Console 是融资演示重点。

展示形式：

```text
Task: 增加用户管理模块

[完成] Planner Agent：已拆解为 5 个子任务
[完成] Codebase Agent：找到路由和页面目录
[运行中] Engineer Agent：正在生成 UserList.tsx
[等待] Reviewer Agent：等待代码生成完成
[等待] Tester Agent：等待生成测试建议
```

真实实现可以简化为：

- 一个任务状态数组。
- 每个阶段调用一个 prompt。
- 不需要复杂并发。
- 不需要真正的多进程 Agent。

Agent Console 最低实现：

| 阶段 | 真实动作 | 展示内容 |
| --- | --- | --- |
| Planner | 调用模型生成计划 | 任务拆解列表 |
| Codebase | 执行代码搜索 | 命中的文件路径 |
| Engineer | 生成 patch 或文件 | 修改摘要 |
| Reviewer | 模板化检查或轻量模型审查 | 风险提示和通过状态 |
| Apply | 用户确认后写文件 | 修改完成状态 |

关键要求：

- Agent 状态必须和实际任务进度同步。
- 不要只做假动画。
- 每个阶段至少产生一条可见日志。
- 失败时显示失败原因和重试入口。

---

### 7.4 Diff Viewer

Diff Viewer 是可信度关键。

需要支持：

- 展示新增文件。
- 展示修改文件。
- 展示删除文件，Alpha 版可以先禁用删除。
- 单文件应用。
- 全部应用。

Alpha 版建议：

- 禁止 AI 自动删除文件。
- 修改前弹确认。
- 应用失败时提示手动复制。
- 优先生成局部 patch。
- 只有小文件才允许整文件重写。

建议限制：

- 超过 300 行的文件默认不走整文件重写。
- 应用前检查原文件是否发生变化。
- Diff 中展示新增、删除和修改行数。
- 现场 Demo 使用小而稳定的演示项目。

## 8. 核心流程设计

### 8.1 打开项目流程

```text
用户选择目录
  → 读取文件树
  → 应用 ignore 规则
  → 生成项目摘要
  → 显示在首页
```

Alpha 版项目摘要可以只基于：

- package.json
- README
- src 目录结构
- 关键配置文件

---

### 8.2 代码问答流程

```text
用户提问
  → 从问题中提取关键词
  → 搜索文件名和文件内容
  → 读取 Top N 相关文件片段
  → 拼接上下文
  → 调用模型
  → 返回答案和引用文件
```

Alpha 版不需要向量数据库。

搜索限制：

- 默认只读取文本文件。
- 单文件最大读取 200KB。
- 单次上下文最多 8-12 个代码片段。
- 默认忽略 `.env`、`.git`、`node_modules`、`dist`、`build`。
- 默认忽略图片、压缩包、数据库文件、lock 文件、source map、minified 文件。
- 搜索结果必须带文件路径。

---

### 8.3 AI 修改代码流程

```text
用户提出修改需求
  → Planner 生成简短计划
  → Search 找相关文件
  → 读取文件内容
  → Engineer 优先生成局部 patch
  → Diff Viewer 展示差异
  → 用户点击 Apply
  → 写入文件
  → 展示完成状态
```

Alpha 版推荐采用“局部 patch 优先 + 小文件整文件重写兜底”的方式。

具体策略：

- 简单 UI 修改：生成局部 patch。
- 新增文件：生成完整文件。
- 小文件修改：允许整文件重写。
- 大文件修改：不允许整文件重写，要求模型输出局部修改。
- patch 失败：展示错误，并允许用户复制建议代码手动处理。

---

### 8.4 一句话生成项目流程

```text
用户描述项目
  → AI 识别项目类型
  → 选择内置模板
  → AI 对模板进行局部定制
  → 写入本地目录
  → 提示安装依赖
  → 给出启动预览命令
```

为了保证演示稳定，可以内置几个模板：

- React 管理后台
- SaaS Landing Page
- Todo App
- AI Chat 页面
- 数据看板 Dashboard

AI 可以基于模板二次修改，这样现场演示更稳定。

Demo 稳定策略：

- 现场优先使用内置模板。
- 模型只负责局部定制，不负责从零生成完整工程。
- 每个模板都提前验证可以安装、启动和构建。
- 如果模型失败，仍可展示模板项目已经生成。

## 9. 最低安全策略

虽然该版本不做企业级安全，但仍建议保留最低安全线，避免演示时误删文件或泄露明显敏感信息。

### 9.1 保留

- 写文件前展示 Diff。
- 执行命令前弹确认。
- 默认不读取 `.env`、私钥、证书文件。
- 默认忽略 `node_modules`、`.git`、`dist`、`build`。
- 禁止 AI 自动删除文件。
- API Key 只存本地，优先使用系统凭据管理器。
- 云端模型调用前提示会发送代码上下文。

### 9.2 暂不做

- 插件签名。
- 沙箱隔离。
- 企业审计日志。
- 团队权限系统。
- 复杂数据出境策略。
- 完整 RBAC。
- Skill 审核系统。
- 远程策略下发。
- 完整交互终端权限系统。

### 9.3 演示版原则

```text
不做重安全，但不做危险自动化。
```

也就是说：

- 可以简化权限系统。
- 可以不做插件市场安全。
- 可以不做企业审计。
- 但不要让 AI 静默删除文件、静默执行命令、静默上传敏感文件。

### 11.1 Demo 兜底方案

现场演示必须准备兜底。

| 风险 | 兜底方式 |
| --- | --- |
| 模型响应慢 | 使用预设模型输出或提前缓存结果 |

| Agent 阶段失败 | Agent Console 使用状态机继续展示过程 |

融资 Demo 的原则：

```text
现场演示一定要稳定，不要把成功率完全交给模型实时发挥。
```

## 16. 结论

开发测试版不需要追求完整安全、完整后端、完整市场和完整企业能力。

最重要的是先做出一个稳定、好看、能现场演示的核心闭环：

```text
打开项目
  → AI 理解代码
  → 用户提出需求
  → 多 Agent 展示执行过程
  → AI 生成修改
  → 展示 Diff
  → 用户应用
  → 项目效果变化
```

这个版本的关键不是“功能多”，而是“看起来像未来的软件开发方式”。

只要这个演示成立，就足够用于早期融资沟通、产品验证和种子用户招募。
